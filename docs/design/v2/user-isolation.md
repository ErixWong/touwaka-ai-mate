## 1. 背景与目标

### 当前问题

所有用户在同一进程中运行，存在安全风险：
- 普通用户可能通过 `execute` 工具读取 `.env` 文件
- 用户 A 理论上可以访问用户 B 的文件
- 敏感配置（数据库连接、API Key）对所有用户可见

### 目标

实现用户级别的隔离，确保：
1. 普通用户无法访问敏感配置（`.env`、数据库连接）
2. 用户之间互相隔离，无法访问对方文件
3. 技能工具在受限环境中执行
4. 技能权限根据角色动态分配

## 2. 用户角色权限

### 2.1 用户角色定义

| 角色 | 技能上限 | 目录范围 | 沙箱上限 |
|------|----------|----------|----------|
| **admin** | Level 4（全部） | `/`（完整访问） | 20 |
| **power_user** | Level 3 | `/skills`（读写）+ `/work/{userId}` | 10 |
| **user** | Level 2 | `/skills`（只读）+ `/work/{userId}` | 5 |
| **guest** | Level 1 | `/work/{userId}`（只读） | 3 |

### 2.2 技能分级

| 级别 | 名称 | 说明 | 示例技能 |
|------|------|------|----------|
| Level 1 | 基础技能 | 只读操作，无风险 | file-reader, web-search |
| Level 2 | 标准技能 | 写入操作，限制目录 | file-writer, skill-developer |
| Level 3 | 高级技能 | 执行命令，有风险 | code-runner, shell-executor |
| Level 4 | 管理技能 | 系统配置，高风险 | skill-manager, system-admin |

详见 [技能权限设计](./tool-permission-design.md)。

## 3. 隔离架构

### 3.1 两层角色模型

```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：用户角色 (User Role)                                  │
│                                                              │
│ 决定：全局权限上限                                            │
│ ├── 目录上限: /work/{user_id}/                              │
│ ├── 技能上限: max_skill_level                               │
│ └── 敏感文件: 始终不可见 (.env, *.key)                       │
└─────────────────────────────────────────────────────────────┘
                            ↓ 继承并收窄
┌─────────────────────────────────────────────────────────────┐
│ 第二层：专家角色 (Expert Role)                                │
│                                                              │
│ 决定：在用户权限内的细分                                      │
│ ├── 目录细分: 只能访问特定 Task 的特定 Phase 目录             │
│ ├── 读写控制: 指定哪些只读、哪些读写                          │
│ └── 技能选择: 从用户可用池中选择                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 沙箱管理

沙箱按需创建，池化管理：

| 场景 | 沙箱类型 | 权限范围 |
|------|----------|----------|
| 用户对话 | dialog 沙箱 | 用户权限全范围 |
| Task 执行 | task-phase 沙箱 | 用户权限 ∩ 专家权限 |

详见 [沙箱架构设计](./sandbox-architecture.md)。

## 4. 敏感文件保护

所有用户角色（包括 admin）的沙箱都禁止直接访问：

| 文件/目录 | 说明 |
|-----------|------|
| `.env` | 环境变量，包含密钥 |
| `*.key` | 私钥文件 |
| `*.pem` | 证书文件 |
| `/etc/shadow` | 系统密码（Linux） |

这些敏感信息只能通过系统 API 访问，且需要相应权限。

## 5. 目录结构

```
/data/
├── work/                      # 用户工作目录
│   ├── user_001/             # 用户 001 的专属目录
│   │   ├── T20260301_001/    # Task 目录
│   │   │   ├── 00-requirements/
│   │   │   ├── 01-analysis/
│   │   │   ├── 02-process/
│   │   │   ├── 03-review/
│   │   │   └── 04-output/
│   │   └── ...
│   └── user_002/             # 用户 002 的专属目录
│       └── ...
│
├── skills/                   # 技能目录（只读或读写视角色而定）
│   ├── builtin/
│   ├── searxng/
│   └── ...
│
└── .env                      # 敏感配置（禁止沙箱访问）
```
