# 专家托管模式

> 创建日期：2026-03-03
> 状态：⏳ 待讨论

## 概述

专家托管模式是一种允许两个专家结对运行、在后台自主完成复杂任务的机制。

## 核心思想

### 双专家协作

- **执行专家（Worker）**：负责实际执行任务、调用工具
- **监督专家（Supervisor）**：
  - 审阅 Worker 的工作成果
  - 提出改进建议和下一步方向
  - 判断任务是否完成或受阻
  - 独占 `shutdown_task` 技能调用权限

### 解决的问题

LLM 调用工具通常有次数限制。对于复杂任务，可能在完成之前次数就用完了。通过双专家多轮对话机制：

1. 监督专家审阅执行专家的工作
2. 提出改进建议或下一步方向
3. 执行专家继续工作
4. 形成循环，直到任务完成

## 关键特性

### 1. 后台运行

- 不需要流式服务（SSE）
- 纯后台异步执行
- 用户可以离开，任务继续运行

### 2. 自主结束机制（Shutdown 技能）

由 **监督专家（Supervisor）** 独自决定是否结束任务，通过调用 `shutdown_task` 技能终止。

**判断逻辑**（Supervisor 自问）：

a. **任务是否已完成？** - 工作目标达成
b. **是否无法继续推进？** - 需要外部资源、需要更多讨论完善需求

满足任一条件时，Supervisor 直接调用 `shutdown_task` 技能。

**技能定义**：

```javascript
{
  name: 'shutdown_task',
  description: '终止当前托管任务（仅 Supervisor 可调用）',
  parameters: {
    reason: '终止原因（completed/blocked/needs_input）',
    summary: '工作总结'
  }
}
```

**对话示例**：

```
[Worker] 执行任务步骤 3，调用工具 C
[Supervisor] 审阅结果，认为任务已完成
[Supervisor] 调用 shutdown_task({ reason: 'completed', summary: '...' })
→ 程序执行技能，停止对话循环
```

**关键点**：
- 只有 Supervisor 有权限调用 shutdown_task
- Worker 不能自行终止，只能等待 Supervisor 决定
- 这是一种"优雅终止"，而非硬性轮次限制
- 结构化的终止方式，便于记录原因和总结

### 3. 定时唤醒

- 设置闹钟，如 1 小时后再检查进度
- 适用于长时间任务的中断恢复

### 3. 多轮对话

```
[Worker] 执行任务步骤 1，调用工具 A
[Supervisor] 审阅结果，建议调整方向
[Worker] 根据建议执行步骤 2，调用工具 B
[Supervisor] 确认进度，建议继续
[Worker] 执行步骤 3...
[Supervisor] 任务已完成，建议结束
→ 托管任务结束
```

```
├─────────────────────────────────────────┤
│  - 创建托管任务                           │
│  - 分配专家配对                           │
│  - 监控任务状态                           │
│  - 处理定时唤醒                           │
└─────────────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│           对话循环引擎                     │
├─────────────────────────────────────────┤
│  1. Worker 执行 → 生成响应               │
│  2. Supervisor 审阅 → 生成建议           │
│  3. 检查结束条件                          │
│  4. 如未结束，回到步骤 1                  │
└─────────────────────────────────────────┘
```

## 架构师视角：方案反思

### 💰 成本考量

| 问题 | 分析 |
|------|------|
| **双倍 LLM 成本** | 每轮对话调用 2 次 LLM（Worker + Supervisor），成本翻倍 |
| **长任务累积** | 复杂任务可能需要数十轮对话，Token 消耗可观 |
| **建议** | 考虑设置消耗上限、使用更便宜的模型做 Supervisor |

### ⏱️ 延迟问题

| 问题 | 分析 |
|------|------|
| **串行执行** | Worker → Supervisor → Worker，每轮都有累积延迟 |
| **用户体验** | 后台任务虽不阻塞用户，但完成时间会显著增加 |
| **建议** | 某些场景可考虑并行审阅（但会增加复杂度） |

### 🔒 可靠性与边界情况

| 问题 | 分析 |
|------|------|
| **Supervisor 判断失误** | 可能过早终止（任务未完成）或迟迟不终止（浪费资源） |
| **Worker 陷入死循环** | 如果 Worker 一直做无用功，Supervisor 能否识别？ |
| **技能调用失败** | shutdown_task 调用失败怎么办？需要重试机制？ |
| **建议** | 增加硬性兜底机制（最大轮次、最大时长、最大 Token） |

### 🏗️ 架构复杂度

| 问题 | 分析 |
|------|------|
| **状态管理** | 需要持久化：任务状态、对话历史、中间结果 |
| **错误恢复** | 任务中断后如何恢复？从哪一轮继续？ |
| **并发控制** | 多个托管任务同时运行，资源如何分配？ |
| **建议** | 设计好状态机，明确各状态转换条件 |

### 📊 可观测性

| 问题 | 分析 |
|------|------|
| **进度透明度** | 用户如何知道任务进行到哪一步了？ |
| **调试困难** | 后台任务出错，如何排查？ |
| **审计需求** | 需要记录完整的对话历史和决策过程 |
| **建议** | 实现详细的日志、进度推送、对话回放功能 |

### 🤔 核心假设的挑战

**假设 1：双专家比单专家更高效**
- ❓ 挑战：真的比给单个专家更多工具调用次数更高效吗？
- 🧪 建议：做 A/B 对比实验

**假设 2：Supervisor 能做出正确判断**
- ❓ 挑战：LLM 本身就有判断失误的可能
- 🧪 建议：设计评估指标，验证 Supervisor 的判断准确率

**假设 3：后台运行是刚需**
- ❓ 挑战：用户真的需要"离开后继续"吗？还是更希望实时参与？
- 🧪 建议：调研真实使用场景

### 🎯 单专家 vs 双专家的本质区别

**单专家模式的困境**：

```
[Expert] 执行任务步骤 1，输出结果
         ↓
         等待输入...谁来给？
         ↓
         ❌ 没有输入，无法进入下一轮
```

单专家需要一个**反思过程**和**持续反馈**，但专家输出完毕后，需要一个输入才能进入下一轮对话。这个输入谁来给？

**双专家模式的核心价值**：

```
[Worker] 执行任务步骤 1，输出结果
         ↓
[Supervisor] 自动提供反馈（审阅 + 建议）
         ↓
[Worker] 收到反馈，继续执行步骤 2
         ↓
         ✅ 形成自给自足的对话循环
```

**结论**：双专家的本质是**自给自足的对话循环**，不需要人类介入就能持续运行。这才是"托管"的核心意义。

### 🔄 单专家自循环方案？

如果坚持单专家，需要解决"谁来给输入"的问题：

| 方案 | 实现方式 | 问题 |
|------|----------|------|
| **系统注入** | 程序自动注入"请继续"或"请反思" | 过于机械，缺乏针对性 |
| **自我反思 Prompt** | 在 System Prompt 中要求专家自问自答 | 效果不稳定，容易跑偏 |
| **工具触发** | 专家调用 `reflect` 工具触发反思 | 仍然需要外部输入来继续 |
| **双角色切换** | 同一个专家切换 Worker/Supervisor 角色 | 本质还是双专家，只是共用一个模型 |

**结论**：单专家模式难以实现真正的"托管"，双专家是更自然的解决方案。
